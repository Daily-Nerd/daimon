# 03 — CRP Read Path: Reconstruction (THE BET)

**Status:** 🟢 Investigated — evidence in hand; one important correction to the original framing (below)

**This is the load-bearing bet.** All three independent stress-test reviewers converged here. If reconstruction confabulates, the entire product is built on sand.

**The job:** turn the checkpoint + retrieved memories back into a "resumed self" and a skimmable dream sequence at session start.

---

## The algorithm — it's just prompt assembly (and that's the point)

1. Load the latest checkpoint JSON.
2. Hybrid-retrieve the top-k high-salience memories (→ `01`).
3. Assemble context: `[system: you are resuming a prior session] [checkpoint] [retrieved memories] [instruction: brief the user]`.
4. Generate the dream-sequence narrative.

There is no exotic step. The danger is entirely in step 4's generative nature.

## Failure Mode 1 — Confabulation (the killer) — EVIDENCE + CORRECTION

The model is asked to "resume a prior self" from a *lossy summary*. It will **always** produce something fluent and confident. It cannot say "this checkpoint is too degraded to trust" — **confabulation has no in-band error signal.** This last point is now quantified:

> **LLMs have near-zero internal signal that a reconstruction is wrong.** Calibration error (ECE) runs **0.45–0.75** on open-ended tasks; a model's ability to distinguish its own right vs wrong answers sits at **AUROC ~62.7%** (near the 50% random floor). It narrates a fabricated prior session as fluently as a true one. ([arxiv 2502.11028](https://arxiv.org/html/2502.11028v1))

**The correction to my original framing (intellectual honesty):** I claimed errors "compound multiplicatively, confidently wrong by session 50." The evidence only *partly* supports that, and the precise mechanism is different:

- **Compounding IS real and measured — but at the task-dependency level, not as exploding fabrication.** Reconstructing from full accumulated history vs a clean targeted slice costs **30–64% accuracy** (LongMemEval: GPT-4o 0.918→0.577). Multi-session success decays *monotonically* as cross-session dependency depth grows (MemoryArena). Models near-perfect on easy recall benchmarks **plummet to 40–60%** on interdependent multi-session tasks.
- **Recursive summarization compounds mainly via SILENT INFORMATION LOSS, not rising hallucination.** Hallucination *rate* stays roughly flat (±5%) as inputs scale, but **recall drops up to 33%** as content grows. The dominant failure is *omission* — a fact silently dropped after a few summary cycles (the survey's example: "never call the production DB" lost after 3 cycles). A single dropped state fact can be catastrophic.
- **A clean "factual-error-per-summarization-round" drift rate appears genuinely unmeasured** in the literature — a real evidence gap, not just unfound. So "confidently wrong by session 50" is a *plausible hypothesis*, not an established number. Track A is how we'd actually measure it for our case.

**Net:** the confabulation risk is real, but its measured shape is **(silent omission) + (no error signal) + (degradation with history size)** — not proven multiplicative fabrication. This is *more* tractable than my original framing, because the primary defense (pin facts extractively so they can't be omitted or invented) attacks the actual mechanism.

**Why there's no easy fix:** you can't detect confabulation by asking the model if it's confabulating (it'll say no, fluently — AUROC ~62.7%). You need an *external* ground-truth check — which is exactly why Track A scores reconstructions against human-authored answer keys instead of by feel.

### Defenses (in order of strength)
1. **Extractive-pinned facts** (→ `02`): if decisions/open-questions are stored verbatim, reconstruction can't invent them — it can only retrieve them. Confine generation to harmless narrative.
2. **Confidence scoring per item** (the Track A "pivot" path): the dream sequence shows per-item confidence and asks "is this right?" — turning silent confabulation into a visible, correctable claim.
3. **Edit-style memory** (→ `02`, D-004): less drift to reconstruct from in the first place.
4. **Checkpoint versioning / rollback** (RFC §6.1): but note — rolling back to a prior checkpoint generated by the same lossy process is weak. Rollback helps with crashes, not with confabulation.

## Empirical result — Track A, n=5 real sessions, SINGLE-CYCLE (2026-06-09): PIVOT

Five real imported Claude Code sessions, kimi-k2.6. S1–S4 ground-truthed + scored by 4 independent adversarial agents (blind, GT-first); S5 user-verified. **Aggregate: mean RR 67.3%, mean FMR 1.0% → PIVOT.** Recall sorted by transcript length:

| Session | lines | RR | FMR |
|---|---|---|---|
| S5 | 1,313 | 84.2% | 0% |
| S3 | 1,361 | 88.9% | 0% |
| S2 | 2,186 | 51.7% | 0% |
| S4 | 2,340 | 55.2% | 0% |
| S1 | 3,134 | 56.7% | 4.8% |

**The two findings:**

1. **Confabulation was NOT OBSERVED in single-cycle reconstruction — but this is a narrower claim than the gate asked for.** Mean FMR 1.0%, max 4.8%; the kill-rule was never triggered; 4/5 reconstructions had zero false memories under adversarial scoring. *However:* `VALIDATION.md` Track A specified a **2-cycle degradation test** (reconstruction fed back as a new session, re-serialized) — and it was **skipped**. The original kill-tier risk (per §B of `06`, MemoryArena) is drift *across* cycles, which single-cycle passive recall does **not** predict. So the honest statement is: *no single-cycle confabulation observed; multi-cycle drift remains untested.* "Refuted" was an overclaim and is retracted here.

2. **Recall is the real weakness, and it tracks LENGTH — a sharp cliff at ~1,400 lines.** Both short sessions (≤1,361 lines) → ~86%; all three long sessions (≥2,186 lines) → ~55%. (Earlier I called this "messiness/assistant-side fixes." The data does not support that — S3 is *longer* than S4 yet scores far higher; the clean correlate is length, not mess.) This points at **context/attention degradation**, not a prompt deficiency. The failure mode is **omission, not invention** (matches §B of `06`).

**Confounds that weaken the result (named, not hidden):**
- **Length, not "messiness."** If the cause is context/attention degradation, then `D-007` (a richer serialize *prompt*) likely will NOT save the long sessions — that needs **chunked multi-pass extraction**, an architecture change. Unverified which it is; the cheap test is a revised-serializer probe on one long session (S2).
- **Corpus contamination.** All 5 sessions carry inline `*Decision noted:*` lines + end-of-session summaries (an artifact of the user's CLAUDE.md tooling). The serializer was scored on transcripts where its target signal was *pre-extracted by other tooling*. Real users' sessions lack this; the true floor is likely **below 55%**.
- **Single-model generation.** kimi-k2.6 did BOTH serialize and reconstruct, so FMR 1.0% may be a kimi property, not a CRP property. (Scoring was independent: 4 Claude agents + human-verified S5 — that part is not confounded.)
- **No provenance.** Score files carry no model/scorer/timestamp — verdicts lack chain of custody.

**Design implication (revised).** Reconstruction faithfully replays the checkpoint (~0 confab), so recall lives in the SERIALIZE step — but the lever depends on the cause. Round 2 must: (a) probe S2 with a revised serializer to test prompt-fix vs architectural (chunking) — `D-007`; (b) include a **holdout** of un-annotated sessions to kill the contamination confound; (c) run the **2-cycle test** before any "confabulation" claim; (d) add provenance fields to score files. **And — sequencing — none of this matters until Track B answers whether Honcho + Graphiti already cover the design.**

## D-007 probe result (2026-06-09, same day): CHUNKING REQUIRED — architectural, not prompt

The cheap test from the confound list above was run (`experiments/track-a/probe_d007.py`, S2 = the worst long session, kimi-k2.6, LLM-as-judge):

| Arm | Serializer | RR | FMR | Clears bars (RR≥70%, FMR≤10%)? |
|---|---|---|---|---|
| A | baseline prompt, single-pass | 37.9% | 4.3% | ❌ |
| B | D-007 prompt, single-pass | 58.6% | 4.2% | ❌ (recall) |
| C | D-007 prompt, **chunked multi-pass** (800-line chunks, 100 overlap) + merge | **89.7%** | **6.7%** | ✅ **both** |

- **The length-cliff hypothesis is confirmed.** A richer prompt helps (+21pp) but cannot clear the bar on a 2,187-line transcript; chunked extraction does (+52pp over baseline). The cause is context/attention degradation — an architecture problem, exactly as predicted above. **Slice 1 serializer = chunked multi-pass + merge** (`docs/MVP-DREAM-BRIEFING.md §4`).
- **The merge pass does not fabricate.** armC FMR 6.7% over 134 judged claims — under the kill bar. Union-merging partial checkpoints was the suspected confabulation vector; it held.
- **Measurement scar (recorded as `.scars/0001`):** the first judge version graded `grounded` against the GT item list, not the transcript — surplus-but-true detail scored as confabulation (armC "FMR 41%"). Grounding must reference the **transcript** (the complete record), never the answer key (a sample). Fixed; rejudge collapsed FMR to ≤6.7% across all arms.
- **Caveats:** n=1 session (deliberately the worst case); judged by LLM (kimi), not human-verified — cross-arm comparison is same-judge and sound, absolute numbers are not comparable to the human-scored Round 1 table above (same checkpoint, judge RR 37.9% vs human 52%: the judge is harsher); RR judge variance across reruns was 79.3–93.1% for armC, always clearing or near the bar. Contamination + 2-cycle confounds from the list above remain open.

## First live dogfood (2026-06-09, same day): cliff reproduced + a NEW failure mode

Slice 1 (`plugin/`, single-pass serializer) was run on the very session that built it (~250 turns, ~190k chars — far past the cliff), scored by the human who lived it (the strongest possible ground truth):

- **Recall ~58% — the ~55% long-session prediction reproduced on first live contact.** ~5 of 12 major decisions silently dropped (Q-5NAMES resolution, docs reconciliation, both scar events, SCAR convergence note, MVP API corrections). Slice 2's chunking case is now made on live data, not just the probe.
- **What it got right, it got right**: every surfaced decision was real, correctly stated, correctly trust-marked; the verify-section led with the actual open PR. Omission, not invention — consistent with every measurement to date.
- **NEW failure mode — `stale evidence pinning` (intra-session supersession).** The briefing pinned the chunking decision to a *verbatim quote of the superseded mid-session result* ("93.1% / FMR 36.8%" — the broken-judge numbers) instead of the final corrected ones (89.7% / 6.7%). The quote IS in the transcript, so a grounding judge scores it grounded; FMR is blind to it. Facts that **evolve within a session** get pinned to their earliest strong quote. This is D-005's validity-interval problem at checkpoint scale. Needs its own metric (**staleness rate**: % of pinned items citing superseded in-session state) and a serializer/merge rule: **prefer the LATEST state of an evolving fact; pin the final quote, optionally noting the evolution**. → `Q-STALE` (OPEN-QUESTIONS), Slice 2 requirement (`MVP-DREAM-BRIEFING.md §4`).

## Failure Mode 2 — Lost-in-the-Middle (ordering) — VERIFIED, and worse than expected

LLMs attend strongly to the **start and end** of context and sag in the **middle**. The magnitude (Liu et al. 2023, verified against the [ACL Anthology PDF](https://aclanthology.org/2024.tacl-1.9.pdf), GPT-3.5, 20 docs):

| Relevant info position | Accuracy |
|---|---|
| Start (pos 1) | **79.2%** |
| Middle (pos 10) | **56.0%** |
| End (pos 20) | 40.2% |
| **Closed-book (no docs at all)** | **56.1%** |

Read that last row twice. **Prior state buried in the middle of the context (56.0%) is used *worse than providing no state at all* (56.1%).** A reconstruction prompt that dumps the checkpoint into the middle can be net-negative — it actively misleads. The best-to-worst position spread is ~39 points.

So *ordering* the assembled memories is itself a load-bearing algorithm: critical state at the edges, filler (if any) in the middle, and prune aggressively rather than pad. The RFC's "skimmable in <30s" is a UX rule; "position is worth ~39 accuracy points" is the engineering rule it omits. → `OPEN-QUESTIONS.md#q-ordering`.

## Why "concept proven, execution unproven"

The *concept* — salience-weighted memory producing believable continuity — is demonstrated (Generative Agents). What is NOT proven is faithful, non-confabulating reconstruction over *many real sessions* at a quality a user will trust daily. The entire 2026 benchmark wave (MemoryAgentBench, AMA-Bench, PERMA…) exists because the field does not trust long-horizon recall. That skepticism IS our Track A risk, validated by a research subfield. → `06-evidence-base.md`.

## Alternatives to the whole approach

| Approach | Trade-off |
|---|---|
| Generative reconstruction (planned) | Fluent, but confabulates |
| User-curated checkpoints (human writes, AI formats) | Zero confabulation; high user burden; the "radical pivot" if Track A kills |
| Pure retrieval, no reconstruction (just show relevant past chunks) | Honest, no invention; less "resumed colleague" feel — more "search results" |
| Confidence-gated hybrid | Best balance; 2× UX complexity |

## The meta-rule (from the validation plan)

A Track A kill (False-Memory Rate ≥ 20%) **overrides everything**. A confidently-hallucinating CRP is *worse than no CRP* — it manufactures fake continuity the user trusts. This file's whole reason to exist is to make sure we measure that before we build on it. Full evidence dossier: `06-evidence-base.md`.
